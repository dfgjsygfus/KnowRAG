# 桌宠双聊天模式正式设计文档

日期：2026-06-08

## 1. 背景

KnowRAG 桌宠当前以桌宠图片作为主要入口。用户点击桌宠后，窗口会展开为完整聊天面板，并通过现有流式聊天接口完成问答。这个模式已经能支撑知识库问答，但对“临时问一句”的场景来说偏重：每次提问都会打开完整聊天窗口，容易打断用户当前工作。

本设计引入两种聊天方式：

- 短聊天：在桌宠旁边打开的小气泡，用于快速问答。
- 长聊天：正常聊天窗口，用于连续对话、查看引用来源和处理较长内容。

两种聊天方式共享同一份会话状态。用户在短聊天中产生的问题和回答，展开到长聊天后必须可见。

## 2. 设计目标

- 让用户可以不打开完整窗口就快速提问。
- 让用户可以从短聊天无缝展开到长聊天，不丢失上下文。
- 保留现有流式回答、问题路由、RAG 引用来源展示能力。
- 第一版尽量复用现有聊天消息结构和流式处理逻辑，降低实现风险。
- 让桌宠优先呈现为轻量助手，复杂工作再进入完整聊天窗口。

## 3. 非目标

- 第一版不做多会话历史。
- 第一版不做会话持久化存储。
- 第一版不新增短聊天专用后端接口。
- 短聊天气泡中不放模型选择、知识库选择、检索参数等重配置。
- 短聊天和长聊天不维护彼此独立的上下文。
- 不在本设计中处理语音、动画、托盘通知和安装包能力。

## 4. 用户体验设计

### 4.1 默认状态

应用启动后只显示桌宠。桌宠继续支持拖拽、右键菜单、缩放和退出等现有行为。

### 4.2 短聊天气泡

用户左键点击桌宠后，桌宠旁边打开一个紧凑聊天气泡。短聊天是默认聊天入口，主要服务于“随手问一句”。

短聊天气泡包含：

- 紧凑消息预览区。
- 输入框。
- 发送按钮。
- 关闭按钮。
- 展开按钮。

短聊天只展示最近一轮或当前正在进行的一轮对话。如果当前没有消息，则展示紧凑空状态。

短聊天支持流式回答。回答过程中，气泡内展示“正在判断问题意图”“正在组织答案”等轻量状态，但不展示完整引用详情卡片。

### 4.3 长聊天窗口

长聊天窗口是完整会话界面，展示共享消息列表中的全部内容，包括短聊天中产生的问题和回答。

长聊天窗口包含：

- 完整消息列表。
- 用户消息和助手消息气泡。
- 流式回答状态。
- 问题路由信息。
- 引用数量。
- 可展开的引用来源详情。
- 问题输入框。
- 关闭或收起控制。

长聊天用于连续多轮问答、阅读长答案、查看引用来源、复制内容和后续扩展会话管理能力。

### 4.4 从短聊天进入长聊天

短聊天气泡中提供“展开”操作。用户点击后：

1. 当前可见界面从短聊天切换为长聊天。
2. 桌宠窗口调整为完整聊天尺寸。
3. 共享消息列表保持不变。
4. 长聊天消息区滚动到最新消息。
5. 如果回答正在流式输出，流式请求继续进行，不重新发起请求。

### 4.5 从长聊天返回桌宠

用户关闭长聊天后，界面返回桌宠默认状态。当前应用会话中的消息仍保留在内存中。

用户再次点击桌宠打开短聊天时，气泡中展示最近对话预览。

## 5. 推荐交互模型

第一版推荐采用以下交互：

- 左键点击桌宠：打开短聊天气泡。
- 短聊天展开按钮：打开长聊天窗口。
- 长聊天关闭按钮：回到桌宠默认状态。
- 右键菜单：保留现有管理台、缩放、退出等操作。
- 可选增强：右键菜单中增加“打开完整聊天”入口。

这个模型让最常见的提问动作保持轻量，同时保留通往完整聊天能力的明确路径。

## 6. UI 状态模型

当前实现中有类似 `expanded` 的展开状态。双聊天模式建议改为更明确的界面状态：

```text
pet
bubble
full
```

状态含义：

- `pet`：只显示桌宠。
- `bubble`：显示短聊天气泡。
- `full`：显示完整长聊天窗口。

回答状态继续独立维护：

```text
idle
thinking
answering
success
empty
error
```

界面状态和回答状态分离后，代码不会把“窗口长什么样”和“后端回答到哪一步”混在一起。

## 7. 数据模型

第一版只维护一份共享消息列表：

```js
messages = [
  {
    id,
    role: "user",
    content
  },
  {
    id,
    role: "assistant",
    content,
    status,
    routeIntent,
    sources,
    errorMessage
  }
]
```

短聊天和长聊天都从这份 `messages` 渲染。

短聊天不复制消息，而是从共享消息中派生预览内容：

- 优先展示最近一轮用户问题和助手回答。
- 如果助手正在回答，展示最近用户问题和正在流式输出的助手消息。
- 如果没有消息，展示紧凑空状态。

从短聊天展开到长聊天时，只切换界面状态和窗口尺寸，不创建新消息，不重复请求。

## 8. 组件边界

### 8.1 应用壳层

应用壳层负责全局状态和跨组件逻辑：

- `chatSurface`
- `messages`
- `question`
- `answerState`
- `abortController`
- 窗口尺寸调整
- 流式事件处理

应用壳层根据 `chatSurface` 决定显示桌宠、短聊天气泡还是长聊天窗口。

### 8.2 桌宠视图

桌宠视图负责：

- 桌宠图片展示。
- 拖拽处理。
- 右键菜单处理。
- 左键点击打开短聊天。

桌宠视图不直接处理流式问答逻辑。

### 8.3 短聊天气泡

短聊天气泡负责：

- 紧凑消息预览。
- 紧凑输入表单。
- 展开操作。
- 关闭操作。

短聊天从应用壳层接收共享消息和回答状态。发送问题时调用共享提交逻辑。

### 8.4 长聊天窗口

长聊天窗口负责：

- 完整消息列表。
- 引用详情展示。
- 完整输入表单。
- 关闭或收起操作。

长聊天同样从应用壳层接收共享消息和回答状态。发送问题时也调用同一套提交逻辑。

### 8.5 共享聊天逻辑

现有聊天辅助函数应继续作为共享逻辑使用：

- `createUserMessage`
- `createAssistantMessage`
- `appendAssistantDelta`
- `updateAssistantMessage`

短聊天和长聊天不能各自定义不兼容的消息结构。

## 9. 窗口尺寸设计

桌宠窗口尺寸应跟随当前界面状态变化：

- `pet`：桌宠独立尺寸。
- `bubble`：桌宠加短聊天气泡尺寸。
- `full`：桌宠加完整聊天面板尺寸，接近当前展开后的窗口尺寸。

窗口调整应继续尽量保持桌宠右下角位置稳定，延续当前体验。

短聊天气泡需要稳定尺寸，避免流式文本、按钮文案或空状态导致窗口频繁跳动。

## 10. 流式问答流程

短聊天和长聊天共用现有 `streamQuestion` 流程。

用户提交问题后：

1. 校验输入非空。
2. 如存在活跃请求，则中止上一条请求。
3. 向共享消息列表添加用户消息和助手临时消息。
4. 设置回答状态为 `thinking`。
5. 调用现有流式聊天接口。
6. 将 `status`、`routing`、`sources`、`delta`、`done`、`error` 事件写入同一个助手消息。
7. 除非用户主动展开或关闭，否则不改变当前界面状态。

如果用户在短聊天流式输出过程中点击展开，当前流式请求继续执行，并更新同一个助手消息。

## 11. 错误处理

- 空问题：前端不发送请求。
- 后端不可用：短聊天展示紧凑错误，长聊天展示完整错误消息。
- 回答已有部分内容后发生错误：保留已有内容，并展示后续错误提示。
- 用户发起新问题导致上一请求中止：不展示错误。
- 未识别的流式事件：忽略，不中断界面。

错误信息写入共享助手消息，保证短聊天和长聊天看到的结果一致。

## 12. 可访问性和交互细节

- 图标按钮需要提供明确的 `aria-label`。
- 短聊天和长聊天都支持 Enter 提交。
- 输入框和消息文本不能溢出容器。
- 关闭和展开操作必须支持键盘访问。
- 短聊天气泡不能完全遮挡桌宠，避免影响拖拽和右键菜单。

## 13. 视觉设计方向

短聊天应像贴在桌宠旁边的轻量对话气泡：

- 紧凑。
- 安静。
- 视觉负担低。
- 不展示完整引用详情。
- 不放复杂设置。

长聊天延续 KnowRAG 完整聊天面板风格：

- 更高的消息阅读空间。
- 清晰的用户和助手消息区分。
- 引用来源按需展开。
- 适合连续工作和长内容阅读。

短聊天不应做成完整聊天窗口的缩小版，否则会失去轻量入口的意义。

## 14. 测试策略

### 14.1 单元测试

需要新增或更新测试覆盖：

- 界面状态转换：`pet -> bubble -> full -> pet`。
- 短聊天预览内容派生逻辑。
- 短聊天提交问题后使用与长聊天一致的消息模型。
- 流式输出过程中展开长聊天不会重复消息。
- 关闭长聊天不会清空当前内存会话。

### 14.2 组件或 DOM 测试

如果当前前端测试环境支持 DOM 断言，应覆盖：

- 点击桌宠打开短聊天气泡。
- 点击气泡展开按钮打开长聊天。
- 长聊天中能看到短聊天产生的消息。
- 引用详情只在长聊天中展示。
- 两种模式下发送按钮禁用状态一致。

### 14.3 手动验收

手动验收步骤：

1. 启动桌宠，确认默认只显示桌宠。
2. 点击桌宠，确认短聊天气泡出现。
3. 在短聊天中提问，确认流式回答正常。
4. 点击展开，确认长聊天中能看到同一轮问题和回答。
5. 在长聊天中继续提问，确认共享会话继续向下追加。
6. 关闭长聊天后重新打开短聊天，确认最近对话预览仍存在。
7. 模拟后端错误，确认两种模式下错误展示一致。

## 15. 验收标准

- 应用具备三种界面状态：只显示桌宠、短聊天气泡、完整长聊天。
- 左键点击桌宠默认打开短聊天，而不是直接打开完整聊天。
- 短聊天可以提交问题并展示流式回答。
- 短聊天可以展开为长聊天。
- 短聊天中产生的消息在长聊天中可见。
- 流式回答过程中展开长聊天不会重启请求，也不会重复消息。
- 长聊天保留现有引用来源展示能力。
- 短聊天不展示完整引用详情卡片。
- 关闭长聊天返回桌宠默认状态，且不清空内存中的消息。
- 现有右键菜单和桌宠缩放行为保持可用。

## 16. 实施阶段

### 阶段 1：状态和布局拆分

- 引入 `chatSurface`，取值为 `pet`、`bubble`、`full`。
- 替换现有对 `expanded` 的直接依赖。
- 更新窗口尺寸工具，支持三种界面尺寸。

### 阶段 2：短聊天气泡

- 新增紧凑气泡 UI。
- 从共享消息列表派生最近消息预览。
- 将气泡输入框接入现有提交逻辑。
- 增加关闭和展开操作。

### 阶段 3：长聊天保留和接入

- 将当前完整聊天面板作为长聊天模式。
- 确保长聊天渲染共享消息列表。
- 保留来源引用、问题路由和错误提示能力。

### 阶段 4：测试和打磨

- 增加状态转换测试。
- 增加短聊天预览测试。
- 做手动 UI 验收。
- 打磨尺寸、换行、按钮状态和空状态文案。

## 17. 后续扩展

第一版保持简单：

- 会话只在当前应用内存中共享。
- 短聊天只展示最近一轮或当前一轮。
- 长聊天主要从短聊天展开进入。

后续可以扩展：

- 多会话历史。
- 会话持久化。
- 右键菜单直接打开完整聊天。
- 短聊天自动收起。
- 长回答在短聊天中自动建议展开。
- 对短聊天和长聊天分别做更细的视觉动效。
