# 文档重复上传防护设计

## 目标

相同内容的文档重复上传时，不再创建新的 SQLite 文档记录，也不再产生可继续入库的新文档对象，从而避免 Milvus 出现重复或残留向量。

## 判定规则

- 使用原始上传内容的 SHA-256 `content_hash` 作为唯一内容标识。
- 文件名或 `source_path` 不同，但内容相同，仍视为重复文档。
- 重复上传返回最早创建的已有文档，HTTP 状态保持 `200`，响应增加 `deduplicated: true`。
- 首次上传响应增加 `deduplicated: false`。

## 数据流

`DocumentRepository.create_or_get_document()` 在 SQLite 写事务内先按 `content_hash` 查询；存在时直接返回已有记录，不存在时创建记录。`document_service` 将仓储层返回的去重状态序列化到上传响应。

管理页收到上传响应后重新加载后端文档列表，避免把已存在文档再次插入前端内存数组。

## 兼容与边界

- 保留现有 `create_document()` 接口，内部复用去重逻辑，避免影响已有调用方。
- 为 `content_hash` 创建普通索引，加速查重；不创建唯一索引，以免已有数据库中的历史重复数据导致启动失败。
- 本次只阻止未来重复上传，不自动删除已有重复记录或 Milvus 历史向量。

## 测试

- 仓储层：相同内容、不同文件名和路径只产生一条记录。
- API 层：第二次上传返回相同文档 ID，并返回 `deduplicated: true`。
- 前端静态约束：上传成功后调用 `loadDocuments()`，而不是直接向列表头部插入响应。
